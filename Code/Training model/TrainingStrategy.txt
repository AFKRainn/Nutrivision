NutriVision — vision model fine-tuning strategy
================================================================================

We are building a single-stage object detector (YOLOv12 via Ultralytics) that reads
fridge and pantry photos and outputs bounding boxes plus ingredient class names.
That list feeds the rest of NutriVision (user corrections, meal suggestions, inventory).
This document is our working plan for data, training, and evaluation. Steps may shift in
practice, but this is what we intend to follow.


What we are training
--------------------------------------------------------------------------------
We load pretrained weights (yolo12n.pt via Ultralytics, trained on COCO) and fine-tune on our data.
We are not training vision from scratch. The network has a backbone (generic visual
features), a neck (multi-scale fusion), and a head (box + class per grid cell). Early
layers keep edge and texture knowledge; the head and upper layers adapt to our classes
and fridge-like scenes.

At inference, an image passes through the network, produces many candidate boxes, then
non-maximum suppression keeps the best detections. During training, the same forward
pass happens, but each candidate is matched to ground-truth labels and scored with a loss.


What one training example is
--------------------------------------------------------------------------------
Per image we need:
- The RGB image (resized at train time, e.g. 640×640).
- A matching .txt label file: one line per object —
  class_id  x_center  y_center  width  height
  with box values normalized to 0–1 relative to image size.
- A data.yaml that lists class names (index → name), nc, and paths to train/val (and
  test if we use it).

Ultralytics reads data.yaml and the folders; we own correctness of paths, class list,
and label files after any merge.


Loss and optimization (what the training loop optimizes)
--------------------------------------------------------------------------------
Each batch: forward pass → match predictions to labels → compute loss → backprop →
optimizer update.

We optimize a weighted sum roughly of:
- Box loss — penalizes wrong location/size (IoU-based; CIoU-style in practice).
- Class loss — penalizes wrong ingredient name (classification on matched cells).
- DFL — refines box boundaries (distribution-based fine regression).

Assignment decides which grid cells are responsible for which ground-truth objects;
unmatched cells are pushed toward background. We do not implement this ourselves;
Ultralytics runs it, but we interpret results through val mAP and visual checks.

We track mAP50 (mean Average Precision at IoU 0.5): for each class, how often
detections are correct and complete, then averaged. Rough targets for us:
usable ~0.5+, good ~0.7+, strong ~0.85+; on a merged community fridge dataset,
~0.6–0.75 is a realistic aim for a first production-quality pass.


Datasets we have under Data/
--------------------------------------------------------------------------------
- aicook — kitchen/fridge-style ingredients; YOLO export.
- Fridge detection — fridge context (layout and yaml to confirm on disk).
- Fridge Object.v3i.yolov12 — fridge objects; YOLOv12 export.
- Fridge objects.v12i.yolov12 — likely overlap with another fridge export; we audit
  before merging to avoid duplicate work.
- FOOD-INGREDIENTS dataset.v4i.yolov12 — large class set (~120); naming may differ
  from other sets.
- LVIS_Fruits_And_Vegetables — broad produce; YOLO layout; strong for fruits/veg.

Several Roboflow-style exports point train/val paths like ../train/images, so images
may live beside dataset folders under Data/, not only inside each subfolder. We trust
disk layout over yaml when paths disagree.


Phase A — Inventory
--------------------------------------------------------------------------------
We list what is under Data/: folder names, presence of data.yaml, where images and
labels actually live, image and label counts, and nc plus class names per yaml.

We record which splits exist (train, val, valid, test) per export and whether val
images are truly held out. This phase is filesystem-first; we do not assume a single
layout across sources.


Phase B — Harmonize classes
--------------------------------------------------------------------------------
We build one canonical class vocabulary for the product that covers every dataset we
merge into training (union or curated superset — we document the choice).

We map synonyms and casing (tomato vs Tomato), slash aliases (LVIS), and spelling
variants (egg vs eggs, tomato vs tomatoes). We document the final name list and per-
dataset old_id → merged_id maps before any merge or train run. Bad harmonization shows
up later as duplicate classes and confused mAP, so we fix it here.

Notebook work includes analysis helpers (proposed_canonical, merge groups) and the
merged vocabulary (MERGED_CLASS_NAMES) built from the union of all yaml sources.


Phase C — Merge and layout (all sources — required before final train)
--------------------------------------------------------------------------------
We produce one YOLO tree: train/val(/test) with images/ and labels/, plus one merged
data.yaml with path, nc, and names. This is the dataset we use for the real fine-tune.

Sources to combine:
- All five YOLO yaml datasets (aicook, FOOD-INGREDIENTS, Fridge Object, Fridge objects
  v12, LVIS_Fruits_And_Vegetables).
- Fridge detection — convert COCO (_annotations.coco.json) to YOLO txt, split train/val.

We remap class IDs per source into the harmonized merged list. We prefix image filenames
by source id when needed to avoid basename clashes. We verify every train/val path
resolves and that each image has a label file (or we handle missing labels explicitly).

The merged tree (Data/merged/ + data.merged.yaml) is the single dataset the final
fine-tune trains on.


Phase D — Fine-tune
--------------------------------------------------------------------------------
The final train runs from the standalone train.py script (not the notebook), pointing
at the merged data.merged.yaml. Running it as a plain Python process (instead of inside
Jupyter) avoids the Windows dataloader-worker spawn problems we hit in the notebook.

Settings we use for the final train (train.py):
- model yolo12n.pt (COCO-pretrained), nc adapted to the merged class count,
- epochs 35, patience 15 (early stop if val mAP plateaus),
- imgsz 640,
- batch 16 on the 16 GB GPU,
- workers 8 (process run, not notebook),
- cache 'disk' (decoded images cached to disk; avoids the RAM-cache + worker-spawn
  memory blow-up we saw),
- cos_lr True, close_mosaic 10,
- device GPU when available,
- project/name -> runs/nutrivision_merged_final.

We hold out a val set that reflects real fridge photos. We watch box loss, cls loss,
and mAP50 on val; we do not treat train loss alone as success.


Phase E — Sanity check before Telegram integration
--------------------------------------------------------------------------------
We run the exported best weights on held-out and a few real fridge photos.

We check: Are common ingredients found? Are boxes plausible? Are confusions clustered
on specific classes (fix labels or harmonization)? We only wire the model into the
bot pipeline after this pass is acceptable for our allowlisted testers.


What Ultralytics does vs what we own
--------------------------------------------------------------------------------
Ultralytics: training loop, assignment, loss, optimizer, checkpoints, default metrics.

We own: dataset choice and merge, class list, path correctness, train/val split,
evaluation on our target domain, and deciding when the detector is good enough for
NutriVision sessions.
